# Кулинарная книга — дизайн-документ (спека)

**Дата:** 2026-06-18
**Статус:** утверждён, готов к составлению плана реализации

---

## 1. Цель и контекст

Перенести рукописную тетрадку с рецептами (которую девушка ведёт со школы) в удобное
цифровое приложение. Приложение должно жить **локально на её айфоне**, работать
**без интернета**, быть бесплатным и не требовать App Store, Mac или аккаунта разработчика.

Пользователь (заказчик) — новичок в программировании. Решения принимаются в пользу
простоты, надёжности и лёгкой расширяемости.

## 2. Кто пользуется

Один человек — девушка заказчика. Технически неподготовленный пользователь.
Интерфейс полностью на **русском языке**. Никакой регистрации, логинов, настройки серверов.

## 3. Ключевые решения (итог брейншторма)

- **Формат:** PWA (Progressive Web App) — устанавливается на домашний экран айфона,
  открывается на весь экран, работает оффлайн. НЕ нативное iOS-приложение.
- **Доставка:** код приложения хостится на **GitHub Pages** (бесплатно, навсегда, https).
  Пользователь один раз открывает ссылку в Safari и добавляет на экран «Домой».
- **Данные:** все рецепты и фото хранятся **только в браузере телефона** (IndexedDB).
  Никуда не передаются. На хостинге лежит лишь «пустая оболочка» приложения.
- **Бэкап:** обязателен. Экспорт/импорт всей коллекции в один файл — защита от потери
  данных (браузер может очистить хранилище) и механизм первичного наполнения.
- **Первичное наполнение:** заказчик присылает фото страниц тетрадки → ассистент
  распознаёт текст и оформляет рецепты → упаковывает в файл импорта → пользователь
  загружает его одной кнопкой.
- **Дизайн (визуал):** выбор стиля и вся «красота» делаются на отдельном этапе
  с помощью скилла `impeccable`. В этой спеке визуальный стиль НЕ фиксируется.

## 4. Архитектура и технологический стек

Одностраничное приложение (SPA), упакованное как PWA. Серверной части нет —
вся логика и хранение на стороне клиента (телефона).

- **React** — UI-библиотека (компонентный подход, большая экосистема, хорошо
  сочетается с `impeccable`).
- **Vite** — сборка и dev-сервер.
- **TypeScript** — типобезопасность, меньше ошибок (полезно при росте проекта).
- **Tailwind CSS** — стилизация (предпочтительный инструмент `impeccable`).
- **Dexie.js** — удобная обёртка над IndexedDB для хранения рецептов и фото.
- **vite-plugin-pwa** — манифест, service worker, оффлайн-кэш «оболочки».
- **React Router** — навигация между экранами.

Рассмотренные альтернативы и почему отклонены:
- *Чистый HTML/CSS/JS* — проще на старте, но при заявленном наборе фишек
  (масштабирование порций, фильтры, режим готовки, реактивные списки) код
  становится трудноподдерживаемым. Отклонено.
- *Svelte/SvelteKit* — хороший вариант, но меньше готовых примеров для новичка
  и чуть менее «накатанная» интеграция с `impeccable`. Отклонено в пользу React.

## 5. Модель данных

Хранилище: IndexedDB через Dexie. Основные сущности:

### Recipe (рецепт)
- `id` — уникальный идентификатор (uuid).
- `title` — название.
- `category` — одна категория (строка из управляемого справочника, напр. «Супы»).
- `tags` — массив строк (напр. `["постное", "праздничное"]`).
- `ingredients` — массив объектов `{ name, amount, unit }`.
  `amount` числовой (для масштабирования порций); если «по вкусу» — `amount = null`.
- `steps` — массив объектов `{ text, durationMin? }`. `durationMin` опционально,
  используется для кнопки-таймера в шаге.
- `photo` — изображение блюда, хранится как Blob в IndexedDB (одно фото в v1).
- `rating` — оценка 0–5 (звёзды).
- `timeMinutes` — общее время готовки (число).
- `servings` — базовое число порций (для масштабирования).
- `notes` — свободный текст (заметки, «от кого рецепт»).
- `isFavorite` — флаг «избранное».
- `createdAt`, `updatedAt` — метки времени.

### Category (категория)
- Справочник категорий, редактируемый пользователем (добавить/переименовать/удалить).
- Предзаполнен разумным набором (Супы, Салаты, Горячее, Выпечка, Десерты, Напитки и т.п.).

### ShoppingItem (позиция списка покупок)
- `id`, `name`, `amount`, `unit`, `checked` (куплено/нет), `fromRecipeId` (опционально).
- Генерируется из выбранных рецептов; можно добавлять позиции вручную.

### Settings (настройки)
- Единицы измерения по умолчанию, прочие пользовательские предпочтения.

## 6. Экраны

- **Главная** — сетка карточек рецептов с фото; строка поиска; фильтры по категории и тегам;
  переключатель «только избранное».
- **Рецепт (просмотр)** — фото, мета (время/порции/оценка), ингредиенты, шаги, заметки;
  кнопки: «Готовить», «В список покупок», управление масштабом порций, «В избранное»,
  «Редактировать», «Удалить».
- **Режим готовки** — экран не гаснет (Wake Lock API); шаги по одному крупным шрифтом;
  свайп между шагами; кнопка таймера у шагов с указанным временем.
- **Добавить / Редактировать рецепт** — форма: название, категория, теги, ингредиенты
  (динамический список), шаги, фото (загрузка с телефона), оценка, время, порции, заметки.
- **Список покупок** — позиции с галочками; добавление вручную; очистка купленного.
- **Настройки** — экспорт/импорт коллекции (бэкап), конвертер единиц, управление категориями.

## 7. Фишки v1 (детально)

- **Масштабирование порций** — множитель = `желаемые_порции / servings`; числовые `amount`
  пересчитываются; «по вкусу» не меняется.
- **Режим готовки** — Wake Lock + пошаговый просмотр + крупный шрифт.
- **Таймеры в шагах** — у шага с `durationMin` кнопка запускает обратный отсчёт;
  по окончании — звук + вибрация (пока приложение открыто; в режиме готовки экран активен).
- **Список покупок** — сбор ингредиентов из выбранных рецептов в единый список; вычёркивание.
- **Избранное + поиск + фильтры** — поиск по названию; фильтры по категории/тегам; избранное.
- **Конвертер единиц** — таблица коэффициентов (граммы ↔ стаканы ↔ ложки, мл ↔ литры и т.п.).
- **Бэкап (экспорт/импорт)** — экспорт всей коллекции в один JSON-файл (фото — в base64
  внутри файла); импорт восстанавливает/наполняет коллекцию. Тот же формат у файла
  первичного наполнения.

## 8. Хранение и бэкап

- Рецепты, категории, список покупок, настройки — в IndexedDB (Dexie).
- Фото — Blob в IndexedDB (для просмотра в приложении); при экспорте конвертируются в base64,
  чтобы вся коллекция помещалась в один переносимый файл.
- Риск: браузер iOS может очистить хранилище при нехватке места/долгом простое.
  Митигация: заметная кнопка бэкапа + мягкое напоминание делать копию.

## 9. Первичное наполнение из тетрадки

1. Заказчик присылает фото страниц тетрадки.
2. Ассистент распознаёт рукописный текст и оформляет в структуру Recipe.
3. Рецепты собираются в файл формата импорта (тот же, что у бэкапа).
4. Пользователь загружает файл в приложении через «Импорт».
5. Замечание: при сложном почерке возможны мелкие ошибки распознавания —
   правятся вручную в приложении.

## 10. Доставка и установка (PWA на айфон)

- Сборка Vite → статические файлы → публикация на GitHub Pages.
- В Vite настраивается корректный `base` под путь GitHub Pages.
- Манифест PWA: имя, иконка, тема, режим `standalone`.
- Service worker кэширует оболочку для оффлайн-работы.
- Установка: открыть ссылку в Safari → «Поделиться» → «На экран Домой».

## 11. Вне рамок v1 (Non-goals)

Сознательно НЕ делаем в первой версии (но архитектуру не ломаем под будущее добавление):
КБЖУ/калории, планировщик меню на неделю, «что приготовить из имеющегося»,
импорт рецепта по ссылке, OCR-сканирование внутри приложения, облачная синхронизация,
шаринг между устройствами, метка сложности.

## 12. Этапы работы

1. **Спека** (этот документ) — готово.
2. **План реализации** — детальный пошаговый план (skill `writing-plans`).
3. **Каркас приложения** — рабочий функционал без финального визуала.
4. **Дизайн** — стиль и красота вместе со скиллом `impeccable`.
5. **Деплой** — публикация на GitHub Pages, установка на айфон, первичное наполнение.

## 13. Риски и открытые вопросы

- Очистка хранилища iOS — закрываем бэкапом (см. §8).
- Качество распознавания почерка — правки вручную (см. §9).
- Имя приложения и иконка — выбираются на этапе дизайна.
- Точный визуальный стиль — выбирается на этапе дизайна с `impeccable`.
