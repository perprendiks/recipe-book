# Облачный автобэкап — дизайн-документ (спека)

**Дата:** 2026-06-18
**Статус:** утверждён, готов к плану реализации
**Базируется на:** готовом приложении «Кулинарная книга» (Фазы 1-2, задеплоено на GitHub Pages)

---

## 1. Цель и контекст

Сейчас бэкап ручной (кнопка «Сохранить копию» → файл). Пользователь (девушка) хочет, чтобы копия данных создавалась и обновлялась **автоматически при каждом изменении**, без ручных действий, и чтобы рецепты можно было **восстановить на новом телефоне**.

Решение: автоматическая синхронизация всей коллекции в облако (Supabase) при каждом изменении, с входом по email/паролю для привязки и восстановления. Облако **опционально** — без входа приложение работает полностью локально, как раньше.

## 2. Ключевые решения

- **Облачный сервис: Supabase** (бесплатный tier). Готовый Auth (email/пароль), Postgres, Row Level Security, простой JS-клиент. Альтернативы (Firebase, свой сервер) отклонены: проще и без вендор-лока.
- **Вход: email + пароль** (Supabase Auth). Надёжно, легко восстановить на любом устройстве.
- **Стратегия синхронизации: last-write-wins** (последняя по времени версия побеждает). Без сложного мержа — для одного пользователя на одном-двух устройствах этого достаточно (YAGNI).
- **Единица синхронизации:** весь бэкап целиком (тот же `BackupFile`, что у ручного экспорта) — один JSON-блоб на пользователя. Переиспользуем существующие `exportBackup`/`importBackup`.
- **Облако опционально:** не вошёл — всё работает локально. Облако включается входом в Настройках.
- **Конфиг Supabase** (URL проекта + anon-ключ) хранится в коде (`src/config.ts`). Anon-ключ публичный по дизайну Supabase — данные защищены RLS и паролем, не самим ключом.

## 3. Архитектура и компоненты

Новые модули (изолированные, с одной ответственностью):

- **`src/config.ts`** — константы `SUPABASE_URL`, `SUPABASE_ANON_KEY`.
- **`src/db/supabase.ts`** — создаёт и экспортирует Supabase-клиент.
- **`src/auth/auth.ts`** — обёртка над Supabase Auth: `signUp(email, pw)`, `signIn(email, pw)`, `signOut()`, `getCurrentUser()`, `onAuthChange(cb)`.
- **`src/db/sync.ts`** — логика синхронизации:
  - `pushBackup()` — `exportBackup()` → upsert в таблицу `backups` (user_id, data, updated_at=now).
  - `pullBackup()` — прочитать ряд пользователя → `importBackup(data, 'replace')`.
  - `syncOnLogin()` — сравнить время облака и локальной коллекции; облако новее или локально пусто → `pullBackup()`, иначе `pushBackup()`.
  - `scheduleSync()` — debounced (≈3 сек) `pushBackup()`; срабатывает только если пользователь вошёл и есть сеть.
  - Статус синхронизации (`idle | syncing | synced | offline | error`) через подписку для UI.
- **UI:**
  - `src/components/AuthSection.tsx` — в Настройках: если не вошёл — форма входа/регистрации; если вошёл — email, статус, кнопка «Выйти».
  - `src/components/SyncStatus.tsx` — ненавязчивый индикатор («✓ Сохранено в облаке» / «Сохраняю…» / «Нет сети»).
- **Триггеры синхронизации:** Dexie-хуки на `recipes` и `categories` (creating/updating/deleting) вызывают `scheduleSync()`. Так ловятся все изменения автоматически, без правки каждого места мутации.

## 4. Модель данных в облаке (Supabase)

Таблица `backups`:
```sql
create table backups (
  user_id uuid primary key references auth.users(id) on delete cascade,
  data jsonb not null,
  updated_at timestamptz not null default now()
);
alter table backups enable row level security;
create policy "own backup only" on backups
  for all using (auth.uid() = user_id) with check (auth.uid() = user_id);
```
Один ряд на пользователя (upsert по `user_id`). RLS гарантирует: пользователь видит и пишет только свой ряд.

## 5. Поток синхронизации

- **При изменении данных** (любая запись в recipes/categories): Dexie-хук → `scheduleSync()` → через ≈3 сек тишины → `pushBackup()` (если вошёл и онлайн).
- **При входе / запуске с активной сессией:** `syncOnLogin()` — сравнить `updated_at` облака с максимальным `updatedAt` локальных рецептов:
  - локально пусто ИЛИ облако новее → `pullBackup()` (replace);
  - иначе → `pushBackup()`.
- **Оффлайн:** `scheduleSync()` тихо пропускается; при восстановлении сети следующее изменение (или вход) до-синхронизирует. Локальная работа не блокируется.
- **Ошибки:** не валят приложение; статус → `error`/`offline`, локальные данные не теряются.

## 6. Безопасность

- Данные привязаны к auth-пользователю; RLS не даёт читать/писать чужой ряд.
- Anon-ключ публичен (дизайн Supabase) — доступ к данным невозможен без аутентификации (email/пароль).
- Репозиторий публичный (GitHub Pages) — это безопасно: в коде только anon-ключ и URL, не секреты.

## 7. UI

- **Настройки → раздел «Облако»:**
  - Не вошёл: поля email/пароль, кнопки «Войти» и «Зарегистрироваться», короткое пояснение про автобэкап.
  - Вошёл: email пользователя, статус синхронизации, время последней синхронизации, кнопка «Выйти».
- **Индикатор статуса** — компактный, в разделе облака (и/или мягко на главной). Без модалок.
- Тёплый стиль существующей дизайн-системы (`.field`, `.btn-primary`, токены).

## 8. Нюансы и ограничения

- **Размер с фото:** фото хранятся как base64 внутри JSON; коллекция с многими фото может стать тяжёлой. Бесплатного tier Supabase (сотни МБ) хватит надолго. Если фоток станут сотни — вынести фото в Supabase Storage отдельно (вне рамок текущей версии).
- **last-write-wins:** при одновременном редактировании на двух устройствах последнее сохранение перезапишет. Для одного пользователя это приемлемо.
- **Зависимость от сети** только для самой синхронизации; приложение остаётся offline-first.

## 9. Что понадобится от пользователя (один раз)

1. Завести бесплатный проект на supabase.com (по пошаговой инструкции).
2. Выполнить готовый SQL (создать таблицу `backups` + RLS) в SQL-редакторе Supabase.
3. Включить email-аутентификацию (по умолчанию включена; при желании отключить подтверждение email для простоты).
4. Передать ассистенту два публичных значения: Project URL и anon public key.

## 10. Вне рамок (Non-goals)

Realtime-синхронизация между двумя устройствами одновременно; CRDT/умный мерж конфликтов; отдельное облачное хранилище фото; шаринг рецептов между пользователями; восстановление пароля по email (полагаемся на стандартный механизм Supabase, отдельный экран не делаем в этой версии).
